# AGENTS.md

- This repo is the app manifest repo inside an nRF Connect SDK west workspace. App code lives under `app/`; the parent workspace contains west-managed `nrf/`, `zephyr/`, `nrfxlib/`, `modules/`, `bootloader/`, and related dependencies.
- Workspace source of truth: the parent `.west/config` points to `nrf/west.yml`, so the active manifest is NCS `sdk-nrf` v2.7.0. This matters for nRF9160 modem APIs such as `nrf_modem_gnss`, which are provided by NCS/nrfxlib and are not available in upstream Zephyr-only workspaces.
- This repo's `west.yml` also imports `sdk-nrf` v2.7.0 for standalone workspace creation. Keep it aligned with the active NCS dependency; do not silently switch back to upstream Zephyr when working on LTE/GNSS.
- `west` is not normally on PATH in this shell. From the parent workspace root, run `source ncs-env.sh` before NCS hardware builds/tests, or use `.venv/bin/west` for simple native commands that do not need the managed toolchain.
- Current implemented app sources are under `app/src/`: `main.c`, `button.c`, `channels.c`, `config.c`, `shell.c`, `state.c`, `led.c`, `lte.c`, `mqtt_client.c`, `sensor.c`, `gnss.c`, and `telemetry.c`, with public headers in `app/include/`.
- Common Zephyr options live in `app/prj.conf`. Native simulator board options live in `app/boards/native_sim.conf` and `app/boards/native_sim_native_64.conf`. nRF9160 DK UART/GPIO/I2C/NVS/LTE/MQTT/GNSS options live in `app/boards/nrf9160dk_nrf9160_ns.conf`; the matching settings partition overlay is `app/boards/nrf9160dk_nrf9160_ns.overlay`.
- Native test command from the parent workspace root: `source ncs-env.sh && west twister -p native_sim/native/64 -T bikeshare-firmware/tests`.
- nRF9160 build command from the parent workspace root: `source ncs-env.sh && west build -b nrf9160dk/nrf9160/ns bikeshare-firmware/app -d build/nrf9160dk -p always`. The older alias `nrf9160dk_nrf9160_ns` still works in NCS v2.7.0 but emits a deprecation warning.
- Implemented shell commands are `bike set id <ID>`, `bike set token <TOKEN>`, `bike set mqtt_host <HOST>`, `bike set mqtt_port <PORT>`, `bike set apn <APN>`, `bike get`, `bike state`, `bike gnss`, `bike lte status`, `bike lte connect`, `bike lte disconnect`, `bike mqtt status`, `bike mqtt connect`, `bike mqtt disconnect`, `bike sim authorize <RENTAL_ID>`, `bike sim cancel [RENTAL_ID]`, and `bike sim button`.
- `state` implements the initial `UNREGISTERED`, `AVAILABLE`, `RESERVED`, and `IN_USE` transitions plus a 60-second reservation timeout. `led` observes state changes and maps them to LED patterns, using `led0` when available. `button` reads `sw0` when available and publishes debounced button events.
- `lte` initializes the nRF9160 modem library, configures APN/PDN, and starts LTE attach. `mqtt_client` can connect and subscribe to the command topic but still needs command payload parsing and telemetry/event publication to the broker. LTE/4G still needs instructor approval if it is intended to satisfy the coursework communication-interface requirement.
- `gnss` is best-effort: on nRF9160 builds with `CONFIG_BIKE_GNSS=y`, it starts direct modem GNSS from background work and caches valid PVT fixes or explicit no-fix status. On native_sim it compiles as an unsupported/no-fix stub. Real GNSS fix acquisition still requires nRF9160 DK hardware, antenna placement, and sky view.
- `telemetry` publishes periodic `telemetry_sample_chan` zbus messages with bike ID, state, uptime, explicit `gnss_fix_valid`, and fixed-point GNSS fields when valid. MQTT serialization/publication of those samples remains pending.
- Twister/ZTEST coverage exists under `tests/` for config validation, core state transitions, LED mapping/cached init, button event publishing, debounce filtering, telemetry formatting, and GNSS valid/no-fix cache transitions.
- Build outputs are expected under parent workspace-level `build/`; VS Code may point IntelliSense at `build/native_sim/compile_commands.json`.
- For every task that changes C code, verify modified code against Zephyr C style guidelines: Linux-kernel-style formatting, 8-character tab stops, snake_case identifiers, 100-column maximum line length, braces on all control-flow bodies, C89 `/* ... */` comments instead of `//`, Doxygen `/** ... */` only for comments meant for generated docs, no binary literals, and no avoidable non-ASCII symbols.
