# Requirements Traceability

This document maps the coursework requirements to the firmware design and implementation.

## Hardware Requirements

| Requirement             | Target implementation                                         | Verification                                                                 | Implementation                                                                                                    |
| ------------------------ | -------------------------------------------------------------- | ----------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Zephyr RTOS application | Standalone Zephyr/NCS C application under `app/`.             | Build with `west build`.                                                     | Builds for `native_sim`, `native_sim/native/64`, and `nrf9160dk/nrf9160/ns`.                                       |
| Debug UART              | Zephyr console, logging, and shell over the board debug UART. | Connect serial terminal and verify boot logs/shell.                          | Console, logging, and shell are configured over the debug UART.                                                    |
| LED                     | Onboard LED through devicetree alias `led0`.                  | State changes produce documented LED patterns.                               | `led` observes bike state and drives the LED pattern through `led0`, with a logical fallback on host simulation.   |
| Button                  | Onboard button through devicetree alias `sw0`.                | Button starts trip from `RESERVED` and ends trip from `IN_USE`.              | `button` reads `sw0`, debounces presses in scheduled work, and publishes button events over zbus.                  |
| Communication interface | LTE/4G modem on nRF9160 DK with MQTT.                         | Device connects over LTE and exchanges MQTT messages with Mosquitto.         | `lte` attaches to the network; `mqtt_client` connects, subscribes to commands, and publishes state/event/telemetry. |
| Optional GNSS           | Best-effort GNSS telemetry on nRF9160.                        | Telemetry includes location when a fix is valid and no-fix status otherwise. | `gnss` runs modem GNSS in the background and caches the latest fix; `telemetry` reports fix or explicit no-fix.    |

## Zephyr Subsystem Requirements

| Requirement | Target implementation                                                                                                                             | Verification                                                                                                                      | Implementation                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| Logging     | Zephyr Logging for boot, config, LTE/MQTT lifecycle, backend commands, state transitions, trips, telemetry publish results, warnings, and errors. | UART logs during demo and tests.                                                                                                  | Boot, configuration, state, LTE/MQTT lifecycle, and command events are logged per the [Logging Policy](#logging-policy). |
| Shell       | Shell commands for setup, diagnostics, and simulated backend rental commands.                                                                     | Run `bike set ...`, `bike get`, `bike state`, `bike lte status`, `bike mqtt status`, `bike sim authorize`, and `bike sim cancel`. | Config, state, LTE/MQTT diagnostic, and simulation commands are implemented.                                             |
| Settings    | Zephyr Settings with NVS backend on hardware.                                                                                                     | Configure settings, reboot, verify values persist.                                                                                | Zephyr Settings with `CONFIG_SETTINGS_RUNTIME` backs `bike/id`, `bike/device_token`, `bike/mqtt_host`, `bike/mqtt_port`, and `bike/apn`. |
| zbus        | Channels: `button_event`, `backend_command`, `bike_state`, and `telemetry_sample`.                                                                | Unit tests and runtime logs show decoupled module communication.                                                                  | All four channels are defined and used: `button` and `mqtt_client` publish, `state` consumes, and `led`/`telemetry`/`mqtt_client` observe state. |
| ZTEST       | Unit tests for state, backend commands, LED mapping, button event flow, button debounce, config validation, and telemetry formatting.             | Run Twister on `native_sim/native/64`.                                                                                            | Suites cover config, state, LED mapping/cached-init, button events/debounce, MQTT helpers, and telemetry/GNSS formatting. |
| Twister     | Automated test execution for non-hardware logic.                                                                                                  | `west twister ...` passes.                                                                                                        | `tests/` runs on `native_sim/native/64`, which avoids hosts missing default native simulator 32-bit runtime libraries.  |

## Functional Requirements

| Requirement          | Target implementation                                                                                                                                   | Verification                                                   | Implementation                                                                                        |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| Bike identification  | Persist `bike/id`.                                                                                                                                          | `bike set id <id>`, reboot, `bike get`.                        | Runtime setting and shell command implemented.                                                          |
| Device token         | Persist `bike/device_token`.                                                                                                                                | `bike set token <token>`, MQTT auth uses token.                | Runtime setting, shell command, and MQTT password use implemented.                                      |
| MQTT broker host     | Persist `bike/mqtt_host`.                                                                                                                                   | `bike set mqtt_host <host>`, MQTT connects.                    | Runtime setting, shell command, validation, and MQTT connection use implemented.                        |
| MQTT broker port     | Persist `bike/mqtt_port`.                                                                                                                                   | `bike set mqtt_port <port>`, validation rejects invalid ports. | Runtime setting, shell command, validation, and MQTT connection use implemented.                        |
| LTE APN              | Persist `bike/apn`.                                                                                                                                          | `bike set apn <apn>`, modem configuration uses APN.            | Runtime setting, shell command, and LTE PDN configuration implemented.                                  |
| Initial state        | Missing/invalid config enters `UNREGISTERED`; valid config and successful initialization enters `AVAILABLE`; unrecoverable init failure enters `ERROR`.   | State-machine tests and boot demo.                             | `UNREGISTERED`/`AVAILABLE` boot rule implemented.                                                       |
| Rental authorization | `RENT_AUTHORIZE` from MQTT or `bike sim authorize <rental_id>` moves `AVAILABLE -> RESERVED`.                                                              | MQTT/demo shell flow.                                          | Shell simulation and MQTT JSON command parsing paths publish through `backend_command_chan`.            |
| Rental cancellation  | `RENT_CANCEL` from MQTT or `bike sim cancel` moves matching `RESERVED` or `IN_USE` rentals to `AVAILABLE`.                                                 | MQTT/demo shell flow.                                          | Shell simulation and MQTT JSON command parsing paths publish through `backend_command_chan`.            |
| Reservation timeout  | `RESERVED` returns to `AVAILABLE` after 60 seconds without button press.                                                                                   | ZTEST with controlled time or manual demo.                     | Implemented with a delayed work item.                                                                   |
| Trip start           | Button press in `RESERVED` moves to `IN_USE`.                                                                                                              | Button demo and state test.                                    | State transition and debounced `button` event publishing implemented; shell can simulate the button.    |
| Trip end             | Button press in `IN_USE` moves to `AVAILABLE`.                                                                                                             | Button demo and state test.                                    | State transition and debounced `button` event publishing implemented; shell can simulate the button.    |
| Invalid commands     | Invalid, duplicate, or state-incompatible backend commands are warned and rejected without state change.                                                  | Backend command tests.                                         | Rejection behavior implemented.                                                                         |
| LED indication       | One LED shows state pattern.                                                                                                                                | Manual demo and LED mapping test.                              | State-to-pattern mapping and `led0` driver path implemented.                                             |
| Telemetry            | Publish periodic telemetry over MQTT.                                                                                                                       | Broker receives messages.                                      | `telemetry` publishes `telemetry_sample_chan`; `mqtt_client` observes the channel and publishes formatted JSON to the telemetry topic. |
| Offline behavior     | Local state continues during LTE/MQTT disconnect; MQTT reconnects in background.                                                                            | Manual disconnect/reconnect validation.                        | Implemented with bounded exponential backoff reconnect; local trip state is unaffected by disconnects.  |

## Logging Policy

Use Zephyr Logging levels consistently:

| Level     | Use                                                                                                           |
| --------- | ------------------------------------------------------------------------------------------------------------- |
| `LOG_INF` | Boot, configuration loaded, LTE/MQTT connected, state transitions, trip start/end, telemetry publish success. |
| `LOG_WRN` | Ignored button presses, invalid backend commands, missing optional GNSS fix, reconnect attempts.              |
| `LOG_ERR` | Settings load failure, modem initialization failure, MQTT setup failure, unrecoverable module errors.         |

## Reference

The project was inspired by the Penn State Hackster project "Bike Share System with Cellular-Based IoT and oneM2M". This firmware design is separate and focused on the Zephyr/NCS coursework deliverable.
